Wale Project Architecture

This document describes the current implementation of Wale as it exists in the codebase today: a Next.js document editor with a TipTap frontend, JSON-file backed document storage, and a backend-owned assistant tool loop.

Current architectural center: the canonical document lives on the backend as a JSON file. The editor syncs with that file, and the assistant reads and edits the document only through backend tools.

High-Level Module Map

The application has three main runtime zones: browser UI, Next.js API routes, and file-backed document services. The browser owns live interaction and rendering. The server owns canonical reads, writes, and assistant tool execution.

Runtime Responsibilities

Routing and App Shell

web/app/page.tsx resolves the current file and tmp query params, redirects to a default document when needed, and mounts the client editor page.

Next.js App Router URL-driven document selection

Editor Runtime

web/app/editor-page-client.tsx creates the TipTap editor with StarterKit and Placeholder, mounts the editor content, and composes the assistant pane beside it.

TipTap Single-page editor workspace

Document Sync

web/lib/document/use-document-sync.ts owns the browser-side sync contract: initial fetch, explicit Cmd+S save, and background polling for external file changes. It now guards against late initial loads overwriting newer local state.

One sync path Canonical backend state

Assistant Transport

web/app/assistant.tsx integrates @assistant-ui/react-ai-sdk. Before sending a user prompt, it saves the current editor JSON to the backend, includes the current document handle, and forwards selection context when available.

assistant-ui Backend-owned document tools

Assistant Execution

web/app/api/chat/route.ts validates requests and passes them to web/lib/assistant/run-assistant.ts, which builds the server prompt, declares tools, and runs the AI SDK streamText tool loop.

AI SDK tool loop Server-only execution

Canonical Document Services

web/lib/document/storage.ts handles safe file IO. web/lib/document/agent-document.ts exposes the assistant-specific read and patch service on top of the stored TipTap JSON, including revision hashing and exact block replacement.

JSON file store Exact patch semantics

Document Read/Write Flow

The editor does not write directly to the filesystem. It talks to the document route, which delegates to the storage module. The same storage module is used by assistant tools, which keeps one canonical persistence path. 

sequenceDiagram
  participant U as User
  participant E as TipTap Editor
  participant S as useDocumentSync
  participant A as /api/document
  participant D as storage.ts
  participant F as JSON file

  U->>E: Type in editor
  Note over E,S: Editor state is live in memory

  alt Initial load
    S->>A: GET ?file=...&tmp=...
    A->>D: readDocument(filename)
    D->>F: read file or create empty doc
    F-->>D: raw JSON
    D-->>A: raw JSON
    A-->>S: JSON document
    S-->>E: setContent(raw)
  end

  alt Explicit save
    U->>E: Cmd+S / Ctrl+S
    E->>S: getJSON()
    S->>A: PUT { content }
    A->>D: writeDocument(filename, content)
    D->>F: write pretty JSON
    F-->>D: ok
    D-->>A: ok
    A-->>S: { ok: true }
  end

  alt External change detection
    S->>A: periodic GET
    A->>D: readDocument(filename)
    D->>F: read raw JSON
    F-->>D: raw JSON
    D-->>A: raw JSON
    A-->>S: raw JSON
    S-->>E: setContent(raw) if changed
  end

Assistant Tool Loop

The assistant uses backend-owned tools immediately. The browser does not apply assistant edits itself. It only saves the current editor state first, then asks the backend assistant to operate on the canonical JSON file. 

sequenceDiagram
  participant U as User
  participant B as Browser
  participant C as /api/chat
  participant R as run-assistant.ts
  participant M as Model
  participant T as document-tools.ts
  participant G as agent-document.ts
  participant F as JSON file

  U->>B: "Rewrite this paragraph"
  B->>F: save current editor JSON via /api/document
  B->>C: POST messages + document handle + selectionText
  C->>R: parse and run assistant
  R->>M: system prompt + messages + read_document/apply_document_edits

  M->>T: call read_document()
  T->>G: readDocumentForAssistant(filename)
  G->>F: read file
  F-->>G: TipTap JSON
  G-->>T: blocks + revision hash
  T-->>M: tool result

  M->>T: call apply_document_edits(baseRevision, edits)
  T->>G: applyDocumentEdits(...)
  G->>F: read file again
  F-->>G: current JSON
  G->>G: validate revision and expectedText
  G->>F: write updated JSON
  F-->>G: ok
  G-->>T: ok + new revision + updated blocks
  T-->>M: tool result

  M-->>R: final assistant text
  R-->>C: UI message stream
  C-->>B: streamed assistant response
  B->>F: document poll sees new JSON
  B-->>B: editor refreshes from canonical state

Design Rules That Shape The Current System

  • There is one canonical document source of truth: the JSON file loaded through storage.ts.
  • The assistant does not patch the editor directly. It edits the backend document through server tools and the editor observes the change through normal sync.
  • Tool edits are exact, not fuzzy. Each edit carries a baseRevision, a blockId, and an expectedText.
  • The current assistant editing surface is intentionally narrow: only top-level paragraphs and headings are exposed as editable blocks.
  • Tests are part of the architecture. The tool layer is validated in isolation before the model is allowed to use it, and the browser flow is validated end to end with a deterministic mock model.

How To Read This Codebase Quickly

If you care about editor behavior

Start at web/app/editor-page-client.tsx, then read web/lib/document/use-document-sync.ts, then web/app/api/document/route.ts.

If you care about assistant behavior

Start at web/app/assistant.tsx, then web/app/api/chat/route.ts, then web/lib/assistant/run-assistant.ts, then web/lib/assistant/document-tools.ts.

If you care about document edits

Read web/lib/document/agent-document.ts and its test file first. That is the exact contract the model relies on.

If you care about confidence

Read web/lib/document/agent-document.test.ts and web/e2e/assistant-document-tools.spec.ts. They show both the isolated tool semantics and the integrated browser flow.
