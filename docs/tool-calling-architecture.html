Tool Calling Architecture

Wale assistant — giving the model document access via server-side vs. client-side tools

The problem

The Wale assistant needs read (and eventually write) access to the document being edited. The document lives in the browser as TipTap editor state. The model runs on an external API (Claude, via the server). How do we bridge that gap?

Key constraint

The TipTap editor is a browser object. Its state only exists in the browser's memory. The server never has it unless the client explicitly sends it.

Current architecture (no tools)

Today, the assistant has no document access. Messages go from the browser to the server, the server attaches a system prompt and forwards to Claude, and the response streams back.

Option A — Server-side tools

The tool is defined on the server with an execute function. When Claude decides to call it, the server runs the function immediately and feeds the result back to Claude. The browser is not involved in the tool execution loop.

The catch

The server doesn't have the document. So the client must pre-send the full document text in every request body. The server tool's execute simply returns that pre-sent text when Claude asks for it.

Server-side

How it works step by step

  1. Client captures editor.getText() at message-send time and includes it in the POST body.
  2. Server defines a read_document tool whose execute returns that text.
  3. Claude sees the tool is available. If it decides to read, it generates a tool call.
  4. The SDK calls execute() on the server immediately — no network hop.
  5. The result is fed back to Claude, which continues generating.
  6. The final response streams to the browser.

Option B — Client-side tools

The tool schema is sent to Claude (through the server), but there is no execute on the server. When Claude calls the tool, the pending call is streamed to the browser, where your client-side handler runs it against the live editor.

Client-side

How it works step by step

  1. Client sends only the message — no document content in the body.
  2. Server defines the tool schema but with no execute.
  3. Claude generates a tool call. The server can't execute it, so it streams the pending call to the browser.
  4. The browser's useAssistantTool handler fires and reads editor.getText() from the live TipTap instance.
  5. The result is sent back to the server, which forwards it to Claude.
  6. Claude generates the final response and it streams back.

Why editing forces the decision

Reading is a nice-to-have where either approach works. Editing is the constraint that determines the architecture. Here's what an edit_document tool looks like in each world:

Server-side edit

Not possible directly

The server returns a structured edit instruction (e.g., "replace paragraph 3 with X"). The client must interpret and apply it after the response completes. The model can't verify the edit worked or make follow-up edits in the same turn.

Breaks the agent loop.

Client-side edit

Natural fit

The tool handler calls editor.commands.insertContentAt() directly. The result (success/failure) flows back to the model. The model can then read again, verify, or make more edits — all in the same turn.

True agent loop.

Recommendation

Go with client-side tools

The document lives in the browser. Both reading and editing it require browser access. Client-side tools give you a single, consistent pattern for all document operations. The extra implementation cost is paid once and reused for every document tool you add.

Server-side tools are the right default for capabilities that live on the server (databases, APIs, secrets). But for anything that touches the in-memory editor, client-side tools are the architecturally honest choice. Mixing the two is fine — they coexist naturally — but the document tools specifically belong on the client.
